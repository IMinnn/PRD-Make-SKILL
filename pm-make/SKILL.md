# PM Make

## 目标

PM Make 用于把粗略产品想法收敛成冻结版产品定型稿（Product Shipping Brief）：一份结构化产品定义文档，通常控制在 1000 个中文字左右或一页以内，可直接用于 AI 编码、原型生成，或作为 PRD Make 工作流的输入。

这个 skill 不是一次性提示词。必须使用工作区式 harness：初始化 6 个聚焦文件，逐个文件填充，暴露开放问题，并且只有在用户确认没有未解决产品决策后，才能冻结定型稿。

## 工作流

1. 创建或复用一个产品工作目录。
2. 如果用户没有提供文档，从 `assets/templates/` 初始化 6 个 Markdown 模板。
3. 将用户原始想法和迭代记录写入 `01-基本描述.md`。
4. 按顺序填充模板。每一轮只聚焦当前文件。
5. 信息缺失时，将精确问题写入 `05-开放问题.md`，不要擅自编造决策。
6. 先只询问阻塞问题。必须在聊天回复中完整列出每个阻塞问题的 ID、简短原因和推荐方案，不能只让用户打开 `05-开放问题.md`。
7. 用户回答阻塞问题后，更新相关文件，并将对应问题转为 `Resolved` 或 `Deferred by user`。
8. 然后在单独一轮对话中询问非阻塞问题。同样必须在聊天回复中完整列出每个问题，并要求用户确认、否定或明确延后推荐方案。
9. 只有当 `05-开放问题.md` 中不存在任何 `Open` 问题，并且用户确认产品方向后，才能冻结。
10. 将最终产品定型稿写入 `06-最终定型稿.md`。

## 六个文件

除非工作区已经存在等价文件，否则使用以下文件名：

- `01-基本描述.md`：原始需求、一句话定位、目标用户、约束、迭代记录。
- `02-JTBD场景.md`：JTBD、Job Stories、ES 风格触发条件场景、场景优先级。
- `03-范围边界.md`：产品形态、MVP 范围、暂缓范围、假设、技术偏好。
- `04-页面流程.md`：页面清单、主流程、状态流转、核心数据对象、权限。
- `05-开放问题.md`：未解决问题、延后决策、已确认决策、风险备注、回答历史。
- `06-最终定型稿.md`：冻结版产品定型稿和变更记录。

## 提问规则

只询问会改变产品行为、工程范围、数据模型、权限、流程或交付优先级的问题。优先提出成组的具体问题，不做泛泛头脑风暴。

对话规则：

- 不允许要求用户打开工作区文件才能回答问题。`05-开放问题.md` 是追踪记录，但聊天回复必须包含当前轮完整问题清单。
- 分两轮提问：先问阻塞问题，再问非阻塞问题。除非用户明确要求完整列表，否则不要把两类问题混在同一轮里。
- 每个问题都要能按 ID 回答。必须包含：问题 ID、需要确认的决策、为什么重要的一句话说明、推荐方案或选项。
- 问题较多时，当前轮只聚焦一个类别，不要一次性倾倒所有开放项。
- 用户回答一轮问题后，先更新 `05-开放问题.md`、已确认决策、延后决策和受影响产品文件，再进入下一轮提问。

状态规则：

- `Open` 只能用于产品仍在澄清中的问题。
- 冻结前，每个问题都必须变成 `Resolved` 或 `Deferred`。
- `Deferred by user` 表示用户明确接受 MVP 推荐默认值，或明确将该主题推迟到 PRD/技术设计阶段。
- 只要 `05-开放问题.md` 中任何行仍为 `Open`，就不能将产品定型稿标记为冻结，包括非阻塞问题。
- 不允许把 skill 自身、模型或“PM Make 默认假设”作为已确认或延后决策的来源。来源必须是用户、用户提供的文档或其他明确外部材料。
- 可以在 `选项/推荐` 中给出建议，但建议不是决策，必须等用户确认后才算数。

需要重点检查的典型维度：

- 输入/来源渠道：手动录入、Excel 导入、API 同步、第三方集成。
- 角色权限：用户角色、管理员角色、主管角色、可见范围边界。
- 通知渠道：站内信、邮件、企微、钉钉、频率、配置粒度。
- 数据模型：核心对象、归属关系、生命周期状态、记录历史。
- AI 配置：模型来源、提示词归属、管理员统一配置、用户级覆盖。
- 日志与审计：记录哪些操作、保留多久、是否导出、合规要求。
- MVP 边界：现在必须做什么、明确暂缓什么、不能隐含什么。

## 冻结标准

只有同时满足以下条件，产品定型稿才能冻结：

- 一句话定位、目标用户、核心场景、MVP 范围、暂缓范围都已明确。
- 核心页面和主流程形成闭环。
- 关键数据对象和权限足够清晰，能让 AI 编码 agent 实现原型。
- `05-开放问题.md` 中所有问题都是 `Resolved` 或 `Deferred`，不存在 `Open` 行。
- 非阻塞问题已由用户回答，或由用户明确延后；不得把模型推荐静默转换成延后决策。
- 每个已确认或延后决策都有用户或文档来源，不能是“PM Make 默认假设”。
- 用户已审阅最终稿，并确认它可以作为 AI 编码或 PRD 生成的唯一输入。

## 最终定型稿格式

最终稿要紧凑、可执行。使用以下结构：

```markdown
# 产品定型稿（Product Shipping Brief）

## 1. 一句话定位

## 2. 目标用户

## 3. 核心场景

## 4. 产品形态

## 5. MVP 范围

## 6. 页面与主流程

## 7. 数据与权限

## 8. 暂缓范围

## 9. 已确认决策
```

不要把定型稿扩写成长 PRD。如果用户要求完整 PRD，切换到 PRD Make，并将冻结版定型稿作为唯一事实来源。
