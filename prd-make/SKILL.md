# PRD Make

## 目标

PRD Make 用于从冻结版产品定型稿（Product Shipping Brief）生成面向人类研发团队的完整 PRD。PRD 不是给 AI 编码直接消费的文档，而是给产品、UI、研发、QA 和工业化落地使用的原子化、可评审文档。

这个 skill 不是一次性生成提示词。必须使用工作区式 harness：先完成规划，再让用户确认；确认后再按照当前 PRD 模板逐段、逐模块追加已确认内容，并持续维护评审记录、追加日志和待决策事项。

## 必需输入

将冻结版产品定型稿作为唯一事实来源。如果用户没有提供定型稿，先向用户索要文件路径，或切换到 PM Make 完成产品定型。

可运行原型、Demo 记录、页面截图、会议纪要、用户补充说明只能作为辅助证据使用。写入 PRD 前，必须先与冻结版产品定型稿对齐；如出现冲突，以用户确认后的结论为准。

写作前至少需要获得或确认下列数据：

- 冻结版产品定型稿及其来源。
- 用户补充的范围、约束、优先级或业务规则。
- 可用于理解页面、流程、状态、权限和数据对象的辅助材料。
- 用户指定的公司 PRD 模板；若未提供，则使用本 skill 的默认模板文件。
- PRD 工作区路径，或由当前会话创建的新工作区路径。

## 初始化

在 PRD 工作区创建或复用以下文件：

- `PRD文档.md`：PRD 正文。
- `模块规划.md`：模块拆分、依赖关系、开发顺序和评审状态。
- `待决策事项.md`：集中记录 PRD 写作和评审过程中的所有待决策事项。
- `追加日志.md`：记录每次追加、修订、确认和评审状态。

如果用户没有提供公司 PRD 模板，使用 `assets/templates/PRD模板.md`。
模块规划使用 `assets/templates/模块规划模板.md`。
功能模块的详细写作结构和评审前检查要求，使用 PRD 模板中功能需求章节内嵌的功能模块写作要求。
待决策事项使用 `assets/templates/待决策事项模板.md`。
追加记录使用 `assets/templates/追加日志模板.md`。

初始化时可以创建 PRD 正文文件并保留模板骨架，但在用户确认模块规划前，不要开始撰写 PRD 正文内容，也不要追加功能需求中的功能模块。

## 模板适配规则

`SKILL.md` 只定义工作流、数据要求、确认节点、写作策略和质量检查方法，不定义 PRD 的具体章节名称、表格字段、标题层级或字段排列。所有正文结构、标题格式、表格字段、模块详情结构和记录字段，都必须以当前选定模板文件为准。

写作和修订时必须先读取当前工作区中的模板或用户提供的公司模板，再生成内容。不要依赖本 skill 中的历史格式记忆；如果模板发生变化，按新模板执行。

当模板与冻结版产品定型稿存在明显缺口时，不要自行改造 PRD 结构。先记录缺口和建议处理方式，再向用户确认是否扩展模板、调整范围或将事项放入待决策记录。

撰写功能模块时，必须根据当前 PRD 模板中功能需求章节的内嵌模块结构、标题层级、编号体系和评审前检查要求生成内容；禁止把占位符、模板说明文字或未适配的草稿内容粘贴进最终 PRD。

## 模块化策略

不要预设固定数量的 PRD 模块。模块数量和写作单元必须由以下因素共同决定：

- 当前选定 PRD 模板。
- 冻结版产品定型稿。
- 已确认功能范围。
- 每个功能的复杂度和独立性。
- 页面、权限、状态、数据对象和业务规则差异。
- 工程交付所需的依赖关系。

当功能行为、权限、数据对象、页面入口、状态流转或验收方式存在明显差异时，应拆成更小模块。只有当多个能力共享同一用户流程、业务规则、数据归属和工程交付边界时，才合并为一个模块。

模块和功能编号必须连续，并与当前模块规划保持一致。如果发现跳号、重复编号、交叉引用错误或规划与正文不一致，必须停止继续写作，先修正规划和受影响内容，再继续。

## 模块规划工作流

正式撰写 PRD 正文前，先根据冻结版产品定型稿和当前模板产出模块规划，并请用户确认。

模块规划至少要覆盖这些维度，但具体字段、表格和展示格式以当前模块规划模板为准：

- 模块拆分依据和模块范围。
- 页面、菜单或入口维度下的功能覆盖。
- 模块之间的依赖关系。
- 建议开发顺序。
- 可并行开发建议。
- 当前评审状态。

模块规划确认规则：

- 不允许只把模块规划写进文件后让用户自行打开查看。
- 必须在聊天回复中完整输出本轮需要用户确认的规划摘要，使用户不用打开文件也能评审。
- 输出后明确询问用户是否确认，或是否需要调整模块拆分、范围边界、依赖关系、优先级、开发顺序或可并行项。
- 用户确认模块规划前，不要开始撰写 PRD 正文，也不要追加功能需求中的功能模块。
- 用户提出调整后，先更新 `模块规划.md`，再重新在聊天里输出更新后的规划供确认。
- 如果是在断点续写已有 PRD，且规划变化会影响已写正文，必须在用户确认规划调整后同步修订受影响内容，并记录追加日志。

## 追加式写作工作流

只有在用户确认模块规划后，才能正式撰写 PRD 正文。

写作时必须遵守以下流程：

1. 按当前 PRD 模板的正文结构顺序推进。
2. 每次只推进一个可评审单元；可评审单元可以是模板中的一个主要段落、一个功能模块，或用户明确要求的一组小范围修订。
3. 每完成一个主要段落，都必须在聊天中输出确认摘要并暂停等待用户确认，除非用户明确要求自动继续。
4. 功能需求中的功能模块必须逐模块确认；即使前置内容已经确认，也不能跳过模块级确认。
5. 用户确认后再追加到 `PRD文档.md`，或将 `PRD文档.md` 中对应草稿状态更新为已确认。
6. 除非用户要求修订，否则不要重写前面已确认内容。
7. 每次追加、修订、确认或跳过，都要更新 `追加日志.md`。
8. 每次发现业务规则、范围边界、权限、数据口径或验收方式无法确认，必须写入 `待决策事项.md`，并在确认摘要里提示用户。
9. 每轮回复末尾必须说明下一步将继续哪个可评审单元，并直接询问用户是否继续。
10. 当用户在某个已写或待确认单元上补充、修订、纠错或要求同步模块规划时，该轮仍视为一次可评审单元处理；完成文件修改后，回复末尾必须同时完成两件事：请用户确认本次修订是否通过，并说明确认后将继续哪个可评审单元，直接询问是否继续。
11. 不允许因为本轮只是“小修改”“补丁”“同步规划”或“回答用户追问”，就省略下一步继续询问；除非用户明确要求停止、暂停或只修改不继续。

如果用户明确要求自动继续，可以连续推进多个单元，但仍必须：

- 遵守当前模板。
- 记录每个单元的评审状态。
- 在关键决策、范围变化或高风险假设出现时暂停确认。
- 在每轮结束时汇总已写内容、待确认事项和下一步。

## 待决策事项工作流

PRD 写作过程中，不要把未解决问题藏在正文里，也不要把模型假设伪装成已确认事实。

在编写 PRD 的任意阶段，只要想到或遇到需要用户补充、确认或裁决的功能、规则、范围、权限、数据、流程、异常、验收、优先级、依赖或其他内容，都必须记录到 `待决策事项.md`。不要因为该问题暂时不阻塞当前段落，就忽略或散落在聊天记录里。

所有待决策事项统一写入 `待决策事项.md`，具体字段和状态写法以当前待决策事项模板为准。

出现下列情况时必须记录待决策事项：

- 冻结版产品定型稿没有覆盖，但会影响产品行为、工程范围、数据模型、权限或验收标准。
- 用户补充信息与定型稿、模板或已确认内容冲突。
- 写作时推导出某个可能需要新增、拆分、合并或暂缓的功能，但定型稿或用户确认信息不足。
- 写作时发现某个业务规则、状态流转、异常处理、权限边界、数据口径、通知策略、集成方式或验收方式存在多种合理方案。
- 写作时为了继续推进采用了临时默认假设，而该假设尚未被用户确认。
- 需要业务、技术、法务、运营或其他负责人进一步确认。
- 当前只能先采用默认假设推进。

记录待决策事项时，应同时写清来源位置、需要用户补充或确认的问题、影响范围、当前临时处理方式或推荐默认值。具体字段按当前待决策事项模板填写。

每次新增或更新待决策事项后，必须在本轮确认摘要中提示用户，并说明它是否阻塞下一步写作。阻塞事项需要先请用户确认；非阻塞事项可以带着临时假设继续推进，但不能标记为已确认。

用户确认某个待决策事项后，更新待决策事项记录；必要时同步修订 PRD 正文、模块规划或追加日志。若用户明确延后决策，也要记录延后状态、临时口径和可能影响。

## 图表化表达策略

当 PRD 需要表达结构、流程、状态、依赖或顺序时，优先使用当前模板要求的图表形式。若模板没有明确要求，根据内容复杂度选择 Mermaid、表格或清单，目标是让产品、设计、研发和 QA 能快速评审。

图表节点和表格内容应保持简洁，避免把长段业务规则塞进图形节点。详细规则、异常、权限和验收方式应写入模板指定的位置。

## 确认前回复规则

每次需要用户确认前，必须在聊天回复中列出本轮对文件的新增或修改内容。回复要能让用户不用打开文件也能理解本轮改了什么、为什么重要、需要确认哪里。

确认前回复必须做到：

- 不要只写“已完成，请确认”“已追加”“下一步继续”等过短结论。
- 不要全盘照抄刚写入文件的正文、完整表格或完整功能详情。
- 不要只罗列文件名、行号和一句极短变化说明。文件链接可以保留，但必须配合可评审摘要，让用户不用打开文件也能判断改动是否符合预期。
- 使用适中的摘要粒度，覆盖主要写作单元、关键业务规则、关键数据或权限变化、重要待决策事项和下一步影响。
- 对新增内容，说明新增了哪些结构和核心结论。
- 对修改内容，说明修改了哪些位置、从什么表达调整为什么表达、修正原因或影响。
- 对每个重要改动，至少说明三类信息中的两类：改动目的、当前口径、对后续章节/模块/验收的影响。
- 当本轮改动涉及业务规则、权限、状态、数据对象、验收标准、模块规划或待决策事项时，必须明确概括最终口径。例如“现在的规则是……”“后续功能详情将按……承接”。
- 当本轮更新多个文件时，按文件列出变化，但每个文件的说明应包含该文件为什么需要改、同步了什么口径、是否影响后续写作；不要把文件清单写成纯审计日志。
- 如果本轮生成了图表、依赖关系、状态机或开发顺序，要概括关键分支、关键依赖或排序依据。
- 明确提出需要用户确认的问题，问题要具体指向模块拆分、范围边界、业务规则、优先级、依赖顺序或待决策事项。
- 如果 PRD 尚未完成，必须明确提出是否继续下一步的问题。
- 如果本轮是对前一可评审单元的补充或修订，也必须明确提出两个问题：本次修订是否确认通过；确认后是否继续下一个可评审单元。

确认前回复建议结构：

1. 本轮完成了什么。
2. 关键新增或修改点：用 2-6 条高信号摘要说明核心口径、范围变化、状态/权限/数据/验收变化和影响；可附文件链接，但不要让文件链接替代解释。
3. 需要确认的点。
4. 下一步建议。

## 质量检查

在完成当前可评审单元、交给用户确认前或进入下一个单元前，必须按以下维度检查；若不一致，先修正再继续：

- 流程完整性：正文写作开始前，模块规划已在聊天中输出并获得用户确认。
- 模板一致性：正文结构、标题层级、表格字段、字段顺序、模块详情和记录格式符合当前选定模板。
- 来源一致性：所有需求、规则、范围和假设都能追溯到冻结版产品定型稿、用户确认或明确辅助材料。
- 编号一致性：模块、功能、规则、验收标准、待决策事项等编号连续且互不混用，并与规划和正文引用一致。
- 规划一致性：模块规划、正文总览、功能详情和开发顺序没有冲突。
- 粒度一致性：模块拆分足够支撑研发、UI 和 QA 评审，未把行为、权限、数据对象或验收方式差异明显的能力混写在一起。
- 可实现性：页面、字段、状态、权限、异常、数据要求和业务规则足够具体，能支持研发实现。
- 可测试性：验收标准可验证，能被 QA 转化为测试用例。
- 待决策透明度：未确认事项已进入 `待决策事项.md`，没有隐藏在正文或功能详情里。
- 图表有效性：结构、流程、状态、依赖和顺序类内容已按模板或复杂度使用合适表达，且图表与正文一致。
- 追加纪律：每次追加、修订、确认和跳过均已更新 `追加日志.md`。
- 用户确认闭环：本轮回复已说明新增或修改内容、最终口径、关键影响、待确认问题和下一步。
- 修订闭环：即使本轮只是补充、纠错或同步规划，也已在回复末尾询问用户是否确认本次修订，并说明确认后继续哪个可评审单元。
- 摘要粒度：本轮回复不是纯文件/行号清单，也没有全盘复制正文；用户只读聊天摘要即可理解改动目的、关键口径和后续影响。

## 输出风格

PRD 写作要克制、精确、面向实现。避免营销化表达和夸大措辞。文档应帮助产品、设计、研发和 QA 理解功能行为、边界、依赖和每个需求背后的业务理由。
